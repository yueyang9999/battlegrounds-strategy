## 第二轮：面板架构重构

### 问题
- 全局重渲染无缓存
- 过滤器状态存储在 DOM class 上（切换后按钮回弹到默认）
- setTimeout hack 处理渲染时序
- 每个页签功能函数平铺，缺少模块化

### 重构方案

**Tab 对象化** — 每个页签是 `{ filtersKey(), render(q) }` 对象：
```javascript
const TABS = {
  heroes: { filtersKey(){...}, render(q){...} },
  minions: { ... },
  powers: { ... },
  // ...
};
```

**过滤器状态外提** — `filterState` 对象独立追踪所有值：
```javascript
const filterState = {
  minions: { tier: 'all', type: 'all' },
  powers: { cat: 'powers', lesser: 'all', mana: 'all', tribe: 'all', tw_tier: 'all' },
  companions: { cat: 'companions' },
  other: { cat: 'anomalies' },
};
```

**视图缓存** — `viewCache[tabId] = { html, filtersKey }`，过滤器/搜索不变时不重渲染。

**Hash 路由** — `#heroes`、`#minions` 等，支持浏览器前进后退和链接收藏。

**事件委托** — 卡片点击、过滤器点击、弹窗联动统一在 content/modal 层级委派，用 `data-cid` 属性传递。

**共享组件** — `cardCellHTML()`、`minionCellHTML()`、`placementBarsHTML()` 消除重复模板代码。

**防抖搜索** — 180ms 延迟，切换页签时清空搜索词但保留过滤器状态。

## 第四轮：阵容编辑器

### team-builder.html（新页面）

模拟酒馆战棋战斗阶段画面：

```
┌──────────┬──────────────────────────────────┐
│ 英雄头像  │  随从1   随从2   ...   随从7     │
│ ❤30 🛡15 │  (130×180 卡片格子)              │
│          │                                  │
│ 英雄技能  │  统计栏:                          │
│ (100×100)│  随从: 5/7  总攻击: 42  总血量: 89│
│          │  种族标签: [龙] [机械]            │
│ 🔹饰品    │                                  │
│ 🔸饰品    │  [导出阵容] [清空全部]             │
├──────────┴──────────────────────────────────┤
│  搜索卡牌...  [英雄] [随从] [技能] ...       │
│  卡牌网格（根据激活格子类型自动过滤）         │
└──────────────────────────────────────────────┘
```

### 核心交互
1. 点击格子 → 高亮激活 → 下方卡牌列表自动过滤兼容类型
2. 点击卡牌 → 填入激活格子 → 自动跳转下一空随从位
3. 选英雄自动填充对应技能（从 hp_ids 取第一个）
4. 点击已填格子的 ✕ 按钮移除
5. 导出阵容 → 生成文本格式并复制到剪贴板
6. 访问：`http://localhost:8765/team-builder`

### 服务端改动
- server.py 新增 `/team-builder` 路由
- `_serve_panel()` 重构为调用 `_serve_page(filename)` 泛型方法

## 架构决策
- 零外部依赖：纯 HTML/CSS/JS，Python 3 stdlib 服务
- 单文件模式：每个页面自包含，不拆分 CSS/JS（保持部署简单）
- JS filterState 为单一真相源：不依赖 DOM class 存储状态
- Hash 路由优于 History API：服务端只需静态文件路由
