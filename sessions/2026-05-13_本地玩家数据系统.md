## 数据流

```
POST /api/games (hero_card_id, placement, ...)
  → PlayerStats.record_game()
    → INSERT INTO game_records

GET /api/player/heroes
  → PlayerStats.get_hero_stats() → SQL GROUP BY
    → CardRegistry 补 name_cn / img
      → JSON 返回

面板: fetchAll() → STORE.playerHeroes / STORE.playerSummary
  → 英雄 tab 切换 mode='personal' → 渲染个人数据表
  → 我的战绩 tab → 渲染统计卡片
```

---

## 关键设计决策

- **至少3场才评最佳英雄**：`get_summary()` 中 best_hero 的 HAVING COUNT(*) >= 3，避免小样本偏误
- **card_type 显式声明**：所有分类 API 必须带 card_type，前端组合使用时依赖此字段过滤
- **单例模式**：PlayerStats 同 CardRegistry/MetaRegistry，使用模块级 get_player_stats()
- **get_card_stats / get_comp_stats**：解析已有 JSON 字段做反向统计，不建物化表

## 经验教训

- 面板端新旧两种数据格式 (playerHeroes vs heroStats) 共存时，需独立的渲染分支。Heroes tab 按 mode 切换表格列结构、数据源。
- API 路径命名 `/api/player/*` 与 `/api/heroes` 等区分，防止前缀匹配冲突
- 并行子 agent 实现多文件改动时，需要我最后做集成验证 — agent 各自完成模块，但交叉引用 (如 server.py 引用 player_stats) 需要确认一致性
